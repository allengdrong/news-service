# 📰 News Service

> 여러 언론사의 최신 뉴스를 자동으로 수집하고,  
> 빠르게 검색할 수 있는 뉴스 검색 서비스

🔗 **서비스 바로가기**  
https://news-service-vatr.onrender.com/

---

## 어떤 서비스인가요?

**News Service**는 여러 언론사의 RSS 뉴스를 자동으로 수집해서  
한 화면에서 최신 기사들을 보여주고, 검색까지 가능한 뉴스 서비스입니다.

매번 포털 사이트를 돌아다니지 않아도,  
실시간으로 업데이트되는 뉴스를 한 곳에서 확인할 수 있도록 만들었습니다.

---

## 주요 기능

### 1. 최신 뉴스 자동 수집
- 언론사의 RSS를 주기적으로 수집합니다.
- 새로운 기사는 자동으로 업데이트됩니다.

---

### 2. 빠른 검색 기능
- 제목 + 요약 기준 검색
- 키워드 기반 기사 조회
- 최신순 정렬

---

### 3. 고성능 검색 엔진
Elasticsearch 기반 검색으로:

- 데이터가 많아져도 빠른 응답 속도
- 전문 검색 최적화

---

## 이런 분들께 좋아요

- 여러 언론사 뉴스를 한 번에 보고 싶은 분
- 키워드로 뉴스 검색이 필요한 분
- 개인 프로젝트 / 포트폴리오 참고용 서비스 찾는 개발자

### Search

## 시스템 아키텍처

```
RSS News Sources
        ↓
Spring Scheduler (RSS Ingestion)
        ↓
Spring Boot Backend
        ↓
Elasticsearch (Search Index)
        ↓
Client Request → Search API
```

## 기술적 강점

### 1. 외부 데이터 수집 및 확장성 고려 설계 (RSS Ingestion)

본 서비스는 여러 언론사의 RSS 피드를 주기적으로 수집하는 구조로 설계되었습니다.
- Spring Scheduler 기반 자동 수집
- 다수 RSS 소스를 공통 인터페이스로 추상화
- 장애 발생 시 개별 소스 실패가 전체 서비스에 영향을 주지 않도록 분리 처리

RSS 수집 로직은 확장 가능 구조로 설계되어, 추가 언론사를 쉽게 붙일 수 있도록 되어 있습니다.

---

### 2. 검색 엔진 중심 아키텍처 (Elasticsearch First)

뉴스 데이터는 InMemory가 아닌 Elasticsearch를 1차 저장소(DB) 겸 검색 엔진으로 사용합니다.
- Full-text 검색 (multi_match)
- 최신순 정렬 (publishedAt desc)
- Upsert 구조로 중복 데이터 자동 처리
- 인덱스 매핑 자동 생성

이를 통해:
- 데이터가 많아져도 성능 저하 최소화
- 검색 품질 향상
- 실무에서 사용하는 검색 구조 그대로 적용

---

### 3. Stateless 서버 설계 (클라우드 친화 구조)

서버는 상태를 전혀 가지지 않는 Stateless 구조로 설계되었습니다.

- 무료 티어를 제공하는 render.com 서비스를 이용했습니다.
- Free tier 서버 환경 특성상 자주 재시작되지만 서비스 영향 없음
- 모든 상태는 외부 저장소(Elasticsearch)에 존재

이는 실제 클라우드 환경에서 요구되는:

>"서버는 언제든 죽을 수 있다" 를 전제로 한 설계 방식입니다.

---

### 4. 환경 분리 설정 구조

설정은 다음 구조로 분리되어 있습니다.
- application.yml → 공통 설정
- application-secret.yml → 로컬 시크릿
- 운영 환경 → 환경변수 기반 주입

이를 통해:
- GitHub에 민감 정보 노출 없음
- 로컬 / 운영 환경 코드 수정 없이 동일 동작
- Render, AWS, 로컬 어디서든 동일 구조로 실행 가능

실무 표준 설정 구조를 그대로 적용했습니다.

---

### 5. Docker 기반 배포 환경

서비스는 Docker 기반으로 패키징되어 배포됩니다.
- 멀티 스테이지 빌드 (build / run 분리)
- JDK 17 기반 경량 이미지
- Render, AWS, naver cloud service 등 어디서든 동일 이미지 사용 가능

이로 인해:
- 환경 차이로 인한 오류 최소화
- 배포 재현성 보장
- 운영 환경과 로컬 환경 일관성 유지

---

### 6. 장애 허용 설계

Elasticsearch 연결 실패 시:
- 자동으로 InMemory Repository로 fallback
- 서비스는 중단되지 않고 제한된 기능으로 계속 동작

즉:

> 검색 엔진 장애 ≠ 서비스 장애

라는 구조로 설계되어 있습니다.

---

### 7. 운영 관점 로그 설계

모든 주요 흐름에 대해 의미 있는 로그를 남깁니다.
- RSS 수집 성공/실패 로그
- Elasticsearch 쿼리/결과 로그
- 사용자 요청 흐름 로그

이를 통해:
- Kibana / Cloud 로그에서 바로 문제 원인 파악 가능
- 실제 운영 환경 디버깅 시나리오 그대로 분석 가능
